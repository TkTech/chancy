chancy.app module¶

class chancy.app.Chancy(*, dsn: str, plugins: list[~chancy.plugin.Plugin] = <factory>, prefix: str = 'chancy_', min_connection_pool_size: int = 1, max_connection_pool_size: int = 10, poll_reconnect_timeout: int = 300, notifications: bool = True, log: ~logging.Logger = <factory>)[source]¶

Bases: object

The main application object for Chancy.

This object is responsible for managing the connection pool to the database, general configuration, and provides some helpers for common tasks like pushing jobs.

async close()[source]¶

Close the connection pool.

Whenever possible, it’s preferred to use the async context manager instead of this method, as it will ensure that the connection pool is properly closed when the block is exited. Ex:

async with Chancy(...) as chancy:
    ...

Declare the queue to the cluster, returning a new Queue with any updated values.

Parameters:

  • queue – The queue to declare.

  • upsert – If set, the queue will be updated if it already exists with the values provided.

async declare_ex(cursor: AsyncCursor, queue: Queue, *, upsert: bool = False) → Queue[source]¶

Declare the queue to the cluster.

This advanced method can be used to declare a queue within an existing transaction.

Note

The cursor given to this function should be a dictionary cursor.

Parameters:

  • cursor – The database cursor to use.

  • queue – The queue to declare.

  • upsert – If set, the queue will be updated if it already exists with the values provided.

async migrate(*, to_version: int | None = None)[source]¶

Migrate the database to the latest schema version.

This convenience method will migrate all plugins that have migration scripts associated with them as well. If you need to migrate a specific plugin, you should call the migrate method on the plugin itself.

If to_version is provided, the database will be migrated to that specific version, up or down as necessary.

async push_many(jobs: list[Job], *, batch_size: int = 1000) → Iterator[Reference][source]¶

Push one or more jobs onto the queue.

Yields a reference for each job pushed after its containing transaction has been committed.

Parameters:

  • jobs – The jobs to push onto the queue.

  • batch_size – The number of jobs to push in a single transaction.

async push_many_ex(cursor: AsyncCursor, jobs: list[Job]) → list[Reference][source]¶

Push one or more jobs onto the queue.

This advanced method can be used to push jobs to the database within an existing transaction. This is very useful when you want to push jobs to the queue only if related objects are successfully created, like running onboarding for a new user only if the user is successfully created.

Parameters:

  • cursor – The database cursor to use.

  • jobs – The jobs to push onto the queue.
